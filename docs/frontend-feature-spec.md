# CookPilot Frontend 기능 명세서

## 1. 문서 목적

이 문서는 CookPilot Flutter 클라이언트의 초기 MVP 기능 범위, 화면 흐름, 사용자 액션, 프론트 상태, 백엔드 연동 필요사항을 정의한다.

CookPilot의 프론트엔드는 레시피 탐색 앱이 아니라, 조리 전-중-후 경험을 연결해 사용자의 입맛과 조리 기록을 누적하는 실시간 AI 조리 코치 경험을 제공하는 것을 목표로 한다.

## 2. MVP 핵심 범위

초기 MVP는 다음 기능에 집중한다.

| 구분 | 기능 | MVP 포함 여부 | 설명 |
| --- | --- | --- | --- |
| 인증 | 로그인 / 게스트 진입 | 포함 | 로그인 없이 탐색과 조리 흐름 진입 가능 |
| 온보딩 | 입맛 프로필 설정 | 포함 | 음식 선호, 매운맛, 기피 재료 기반 콜드스타트 |
| 탐색 | 홈 / 검색 | 포함 | 레시피 추천, 최근 조리, 즐겨찾기, 키워드 검색 |
| 레시피 | 상세 / 버전 확인 | 포함 | 기본 레시피와 나 맞춤 기록 확인 |
| 조리 준비 | 조리 설정 | 포함 | 인분, 재료 수정, 생략, 대체, 양 조절 |
| 조리 중 | 단계 안내 / 타이머 / 음성 코치 | 포함 | 손을 덜 쓰는 조리 흐름 지원 |
| 조리 후 | 리뷰 / 변경사항 확인 | 포함 | 만족도, 메모, 자동 기록된 변경사항 확인 |
| 메모리 | 나 맞춤 / 변형 저장 | 포함 | 다음 조리에 반영할 저장 방식 선택 |
| 확장 | 냉장고 / 장보기 / 커뮤니티 | 제외 | MVP 검증 후 백로그로 분리 |

## 4. 사용자 주요 흐름

### 4.1 게스트 조리 흐름

1. 사용자가 앱에 진입한다.
2. `게스트로 둘러보기`를 선택한다.
3. 홈에서 추천 레시피 또는 검색 결과를 선택한다.
4. 레시피 상세에서 재료, 조리 순서, 기존 나 맞춤 기록을 확인한다.
5. 조리 설정에서 인분과 재료 변경을 확정한다.
6. 조리 중 화면에서 단계 안내, 타이머, 음성 질문 안내를 사용한다.
7. 조리 완료 후 만족도와 메모를 입력한다.
8. 레시피 메모리에 저장하려는 순간 로그인을 유도한다.  
   현재 MVP mock에서는 로그인 유도 화면은 별도 구현 전이며, 실제 저장 흐름으로 바로 연결한다.

### 4.2 회원 조리 흐름

1. 사용자가 소셜 로그인 또는 이메일 로그인을 선택한다.
2. 신규 회원은 입맛 프로필을 설정한다.
3. 홈 추천과 검색 결과에 입맛 프로필이 반영된다.
4. 조리 완료 후 `나 맞춤 업데이트` 또는 `새 변형 저장`을 선택한다.
5. 레시피 메모리에서 저장된 버전을 다시 조리할 수 있다.

### 4.3 반복 조리 개인화 흐름

1. 사용자가 동일 레시피를 다시 선택한다.
2. 상세 화면에서 이전 기록 요약을 확인한다.
3. 조리 설정 화면에서 나 맞춤 버전을 기본으로 사용한다.
4. 조리 후 변경사항이 누적된다.
5. 다음 조리 시 간, 시간, 순서 추천이 더 구체화된다.

## 6. 프론트 상태 모델

초기 구현은 mock 데이터 기반이지만, 실제 구현 시 다음 상태가 필요하다.

| 상태 | 설명 |
| --- | --- |
| `authState` | 게스트 / 로그인 / 신규 회원 여부 |
| `tasteProfile` | 선호 음식, 매운맛, 기피 재료, 알레르기 |
| `recipeSearchState` | 검색어, 필터, 결과 목록 |
| `selectedRecipe` | 상세/조리 대상 레시피 |
| `cookSetupState` | 버전, 인분, 재료 수정사항 |
| `cookSessionState` | 현재 단계, 타이머, 일시정지, 자동 저장 시각 |
| `reviewState` | 별점, 메모, 변경사항, 사진 |
| `recipeMemoryState` | 저장된 버전, 나 맞춤 여부, 변형 목록 |

## 7. 백엔드/API 연동 필요사항

### 7.1 인증

| API | 목적 |
| --- | --- |
| `POST /auth/login` | 이메일 로그인 |
| `POST /auth/social/{provider}` | 카카오/Google 로그인 |
| `POST /auth/guest/convert` | 게스트 기록을 회원 계정으로 이관 |

### 7.2 사용자 프로필

| API | 목적 |
| --- | --- |
| `GET /users/me/profile` | 내 입맛 프로필 조회 |
| `PUT /users/me/profile` | 입맛 프로필 저장 |

### 7.3 레시피

| API | 목적 |
| --- | --- |
| `GET /recipes` | 검색/필터 기반 목록 조회 |
| `GET /recipes/recommended` | 홈 추천 레시피 조회 |
| `GET /recipes/{recipeId}` | 레시피 상세 조회 |
| `POST /recipes/{recipeId}/favorite` | 즐겨찾기 |

### 7.4 조리 세션

| API | 목적 |
| --- | --- |
| `POST /cook-sessions` | 조리 세션 시작 |
| `PATCH /cook-sessions/{sessionId}` | 현재 단계, 타이머, 변경사항 자동 저장 |
| `POST /cook-sessions/{sessionId}/voice-query` | 음성 질문 텍스트 전달 및 코치 응답 수신 |
| `POST /cook-sessions/{sessionId}/complete` | 조리 완료 |

### 7.5 레시피 메모리

| API | 목적 |
| --- | --- |
| `GET /recipe-memories` | 저장된 레시피 메모리 목록 |
| `GET /recipes/{recipeId}/memories` | 특정 레시피 버전 목록 |
| `POST /recipes/{recipeId}/memories` | 나 맞춤 또는 변형 저장 |
| `PATCH /recipe-memories/{memoryId}` | 저장된 버전 수정 |

## 8. 비기능 요구사항

| 항목 | 요구사항 |
| --- | --- |
| 사용성 | 조리 중 한 손 조작과 큰 터치 영역 우선 |
| 접근성 | 주요 버튼 48dp 이상, 텍스트 대비 확보 |
| 안정성 | 조리 세션 자동 저장 |
| 지연 시간 | 음성 질문 응답은 체감상 2~3초 이내 목표 |
| 오프라인 대응 | 조리 중 타이머와 현재 단계는 로컬 유지 |
| 개인정보 | 알레르기, 식단 정보는 명시적 동의 후 저장 |
| 안전 | 식품안전/알레르기 관련 응답은 보수적 안내 |

## 10. PR 분리 제안

초기에는 화면 단위 PR보다 하나의 MVP PR 안에서 커밋을 작게 나누는 방식을 권장한다.

| PR | 내용 |
| --- | --- |
| `feat : Flutter MVP 초기 화면 구현` | 현재 브랜치. 프로젝트 생성, 테마, mock 데이터, 화면 흐름 |
| `feat : 프론트 상태관리 구조 도입` | Riverpod 등 상태관리 추가 |
| `feat : 레시피 API 연동` | 검색/상세/추천 API 연결 |
| `feat : 조리 세션 API 연동` | 세션 시작, 단계 저장, 리뷰 저장 |
| `feat : 음성 코치 연동` | STT/TTS, AI 질의응답 |

## 11. 완료 기준

MVP 프론트 1차 완료 기준은 다음과 같다.

- 사용자가 로그인 또는 게스트로 홈에 진입할 수 있다.
- 검색/홈에서 레시피 상세로 이동할 수 있다.
- 상세에서 조리 설정을 거쳐 조리 중 화면으로 이동할 수 있다.
- 조리 중 단계 진행 후 리뷰와 저장 방식 선택까지 도달할 수 있다.
- 레시피 메모리에서 저장된 버전을 확인하고 다시 조리할 수 있다.
- `flutter analyze`가 통과한다.
- Android 에뮬레이터에서 debug 실행이 가능하다.
